# Agent to Skill 修改方法总结

修改方法可以概括成一句话：

**保留原 agent 的专业知识和工作方法，删除角色扮演式人设，把文件改造成更像可调用的 Skill。**

## 1. 保留主体内容，不重写专业方法论

迁移时不直接推翻原文件正文，而是尽量保留这些部分：

- 核心使命
- 关键规则
- 技术交付物
- 工作流程
- 模板、清单、示例

这些内容是原 agent 真正有价值的部分，也是 Skill 继续可用的知识主体。

## 2. 在文件前面补一层 Skill 接口

给每个 `SKILL.md` 增加统一的最小 Skill 头部，让它从“角色说明”变成“可调用说明”。

统一补的字段是：

- `适用场景`
- `所需输入`
- `交付产物`
- `约束与边界`
- `组合方式`

这一步的目标是让 Skill 一开始就能回答：

- 什么时候该用它
- 调它之前要准备什么
- 它会产出什么
- 它不该怎么被用
- 它能否和其他 Skill 协同

## 3. 把强 agent 口吻改成中性领域说明

重点处理原来这类内容：

- “你是……专家”
- “你的身份与记忆”
- “你记得……”
- “你看过……”
- “你经历过……”
- “你的工作就是……”

这些句子对 agent 有用，对 Skill 不合适。处理原则不是简单全删，而是做“信息保留、口吻去人设”的改写。

主要规则是：

### 标题改写

- `身份与记忆`
- `你的身份与记忆`
- `领域背景与工作假设`

统一收敛成：

- `领域背景`

### 标签改写

- `角色` -> `专业定位`
- `个性/性格` -> `工作方式`
- `记忆` -> `重点关注`
- `经验` -> `典型经验`

### 句式改写

- `你记得……` -> `重点关注……`
- `你熟记……` -> `重点覆盖……`
- `你看过/见过/经历过……` -> `熟悉/覆盖……`
- `你是一位……` -> 改成客观陈述

### 删除纯表演性质段落

例如整段“你是一位……你知道……你的价值在于……”这类开场白，应删除或压缩成中性描述。

核心原则是：

**保留领域判断，删除角色代入。**

## 4. 按目录语境修正模板，而不是一套模板强灌所有文件

不同目录不能硬套同一套头部模板。

### `design/`

改成设计语境：

- 输入改成品牌资产、视觉参考、受众、场景、尺寸等
- 输出改成设计方向、视觉/交互/研究建议

### 其他目录

保留各自更接近原始用途的模板：

- 工程类偏技术方案
- 业务类偏流程、策略、协同
- `marketing` 偏渠道、内容、转化
- workflow 类偏步骤、工具、回流闭环

也就是说，方法不是“完全统一”，而是：

**统一结构，不统一领域语义。**

## 5. frontmatter 收敛到最小必要集合

先把 frontmatter 收敛成最小稳定集合：

- `name`
- `description`

这样做的目的，是让 Skill 元数据先统一，再逐步补其他需要的展示层字段。

## 6. 当前仍需注意的问题

这轮修改完成后，结构已经更接近标准 Skill，但并不意味着所有字段都已经是最终形态。

例如：

- `name` 当前仍可能是中文显示名
- 从标准 Skill 角度，后续应改成 slug，例如 `marketing-app-store-optimizer`

所以这一步是：

- 先统一字段结构

而不是：

- 最终格式完全正确

## 7. 修改原则

如果抽象成方法论，核心有四条：

- 保留知识，不保留人设
- 保留方法，不保留表演
- 先补 Skill 接口，再清理口吻
- 统一结构，但尊重不同领域语境

## 8. 一句话总结

这次不是“把 agent 全部重写成新文档”，而是用最小改动，把原 agent 从“角色提示词”逐步收敛成“标准 Skill 说明”。
